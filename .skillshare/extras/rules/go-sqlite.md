# Go ‚Äî SQLite Patterns & Query Optimization

## Connection Management

- **SQLite read/write connection separation.** `Store` has two DB pools: `db` (write, MaxOpenConns=1) and `rdb` (read, MaxOpenConns=4). All `Query`/`QueryRow` calls must use `s.rdb`; all `Exec`/`Begin` mutations must use `s.db`. Both pools must share the same DSN with identical `_pragma` parameters (busy_timeout, journal_mode) ‚Äî otherwise one pool silently lacks retry or WAL behavior. WAL mode enables concurrent reads while a write TX is active ‚Äî using the wrong pool re-introduces the single-connection bottleneck that blocks API reads during scans.
- **Batch long write transactions.** `RecordScan` splits inserts into batched TXs (~500 assets per batch) with a `status='recording'` ‚Üí `'completed'` lifecycle. Never wrap thousands of inserts in a single TX ‚Äî it holds the SQLite write lock for the entire duration, starving other write operations. On failure, `DELETE FROM scans WHERE id=?` cascades cleanup via FK constraints.

## Query Patterns

- **SQL filter builders must return `(string, []any)`, never bare `string`.** Functions that construct WHERE fragments with dynamic values (LIKE patterns, regex, JSON paths) must return the SQL string together with bind parameters. A function returning only `string` forces callers to embed values via `fmt.Sprintf` or manual `strings.ReplaceAll` escaping ‚Äî both bypass `database/sql` parameterization and create injection risk. The pattern is `func xxxSQL(...) (string, []any)` with `?` placeholders in the SQL and values in the args slice.
- **Store-layer functions must validate inputs independently of handlers.** Even when an HTTP handler sanitizes a parameter (e.g. `sanitizeLocale` whitelist), the Store function that interpolates it into SQL must re-validate. Store functions can be called from multiple code paths (handlers, background jobs, tests, future refactors) ‚Äî trusting the caller is a single point of failure. Use a shared validator (e.g. `validLocaleOrEmpty`) at the Store function entry point and return early (empty result, not error) for invalid values.

## Performance

- **Merge homogeneous COUNT queries into a single CASE WHEN.** When multiple queries scan the same base table with the same WHERE clause but different aggregation conditions (e.g. `COUNT` with different `EXISTS` subqueries), merge them into one query with `COUNT(CASE WHEN <condition> THEN 1 END)`. Each eliminated query saves a full WHERE clause evaluation and table scan. Applied pattern: 3√ó `catalogItemTotalForStatus` ‚Üí 1√ó `catalogOptimizationStatusTotals`, 2√ó EXIF `QueryRow` ‚Üí 1√ó combined.
- **Use correlated EXISTS instead of LEFT JOIN (SELECT DISTINCT ...) in SQLite.** SQLite materializes LEFT JOIN subqueries into temporary hash tables. A `LEFT JOIN (SELECT DISTINCT col FROM big_table WHERE ...) sub ON sub.col = a.col` builds the full distinct set in memory before joining. Replace with `EXISTS (SELECT 1 FROM big_table WHERE scan_id = a.scan_id AND asset_id = a.asset_id)` ‚Äî it uses the index for a single seek per row, avoids temporary table creation, and reduces both memory and CPU. Also reduces the number of bind parameters (no need to pass `scanID` separately to the subquery).
- **Split OR on different indexed columns into separate EXISTS.** SQLite cannot use two indexes in a single correlated subquery with `WHERE col_a = ? OR col_b = ?` ‚Äî it picks one index or falls back to a full scan. Split into `EXISTS (... AND col_a = ?) OR EXISTS (... AND col_b = ?)` so each subquery hits its own index. This applies to `near_duplicate_snapshots` (indexes on `(scan_id, left_id)` and `(scan_id, right_id)`) and any future table with the same pattern. The cost multiplies with `catalogItemFacets`, which runs 12+ queries per API call ‚Äî a single slow subquery becomes a multi-second API freeze.
- **Batch read queries with IN clause, never per-item QueryRow loops.** Functions that load results for a list of items (AI tags, OCR, similar items) must use `inClauseSQL()` + single `Query()` + map-back ‚Äî the same pattern as `hydrateAssetOptimization` in `catalog_batch.go`. A per-item `QueryRow` loop is an N+1 that scales linearly with batch size. Use `contentHashes(items)` to extract deduplicated content hashes for `IN (?)` clauses against `ai_tags` / `ocr_results`.
- **Never apply SQL functions on indexed columns in WHERE clauses.** `LOWER(a.ext)`, `UPPER(a.name)`, or any function wrapper on a column prevents SQLite from using the index on that column. The scanner already stores `ext` as lowercase (`strings.ToLower`), so `LOWER(a.ext)` is both redundant and harmful. If case-insensitive comparison is needed and the column isn't pre-normalized, add an expression index (`CREATE INDEX ... ON table(LOWER(col))`) instead.
- **Skip facets on paginated requests (offset > 0).** Frontend only reads facets from `pages[0]` (`firstPage?.facets`). Catalog API handlers (`CatalogItems`, `CatalogLintFindings`) must check `offset == 0` before calling facet functions ‚Äî otherwise every scroll-triggered page load re-executes 13+ facet queries that the frontend ignores. `CatalogDuplicates` already does this correctly.
