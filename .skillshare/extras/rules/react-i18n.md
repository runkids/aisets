# React / TypeScript — i18n Rules

- **i18n code-first:** Backend API must return machine-readable codes (`reasonCode`, `code`, `suggestionCode`), not hardcoded human-readable strings. Frontend translates codes via `t(\`namespace.${code}\`, { defaultValue: fallbackString })`. Never display raw backend English strings to the user.
- **i18n key hygiene:** Only add locale keys for namespaces that the UI actually reads, or for backend machine codes that can flow into that namespace. Remove unused translation keys instead of keeping speculative duplicates — stale keys hide missing-key regressions and create unnecessary translation maintenance.
- **i18n CJK punctuation:** zh-TW / zh-CN / ja / ko translations must not end with a period (句號 `。`). Use bare sentences without trailing punctuation. English locale keeps standard punctuation.
- **i18n required locale scope:** Every new or modified i18n key must include English. Add non-English locale entries only for the display language(s) the change actually surfaces or the user asks for; do not bulk-fill all five locales speculatively. Missing inactive locale keys should fall back to English until that locale is translated through the normal workflow.
- **English translation views use raw fallback, never blanking.** When a UI surface is showing translated AI category/tag text in English (`viewLocale === "en"` or app locale `en`), render the English translation when available and otherwise fall back to the raw stored label. Do not hide untranslated non-English labels and do not append `(${raw})` in English mode — users should still see a stable label even when translation coverage is incomplete.
- **Localized catalog surfaces must request locale-aware data.** Browse cards, drawers, category dropdowns, and tag editors that display AI category/tag translations must pass the current app locale (`i18n.language`) to their catalog/detail APIs. Do not assume render-time translation is enough — these surfaces depend on backend-provided `categoryI18n`, `tagsI18n`, or translated facet maps, so omitting `lang` silently leaves parts of the UI in the default language.
- **i18n for lint messages:** Use `t(\`lint.rule.${finding.ruleId}.message\`, { defaultValue: finding.message, ...extractedArgs })`. When messages contain dynamic values (KB, paths), extract them on the frontend via regex from the English fallback string. Never rely on unpersisted `args` fields.
- **i18n tab label length:** English labels in horizontal `<Tabs>` groups are often 2–3× wider than CJK equivalents ("Dimensions" vs "尺寸"). When multiple `<Tabs>` groups share one row, keep English labels to 3–6 characters (e.g. "Fmt", "Dims", "Crit", "Warn", "Suggest"). Always verify the English locale at 1440px after adding or renaming tabs — overflow that doesn't appear in zh-TW will break English.
- **Editable tag UIs show localized labels but mutate raw tags.** Tag chips in drawers, popovers, and similar editors may display `tagsI18n[locale]`, but add/remove operations must still read and write the raw `tags_json` values. Never submit translated labels back to the mutation API — translations are presentation-only and the stored tag identity remains the raw string.
